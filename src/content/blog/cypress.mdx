## ما هو Cypress؟

**Cypress** هو **Testing Framework** من الجيل الجديد، مبني خصيصاً لاختبار تطبيقات الويب الحديثة باستخدام **JavaScript**.
فكرته إنه يسهّل على الفِرق كتابة الـ **Test Cases** وصيانتها بدون تعقيد، ويكون جزء طبيعي من سير العمل سواء في الـ **Development** أو الـ **QA**.

---

### من يستخدم Cypress؟

- **Developers** - لكتابة اختبارات مع الكود مباشرة
- **QA Engineers** - لتغطية الـ **Test Scenarios** بشكل منظم
- أي فريق يبغى يرفع جودة تطبيقاته ويضمن ما فيه **Regression** في كل **Release**

---

### ماذا يقدم لك Cypress؟

| النوع                     | الوصف                                                               |
| ------------------------- | ------------------------------------------------------------------- |
| **End-to-end Testing**    | اختبار التطبيق كامل من البداية للنهاية كما يستخدمه المستخدم الحقيقي |
| **Component Testing**     | اختبار مكوّنات واجهة المستخدم بشكل منعزل                            |
| **Accessibility Testing** | التحقق من توافق التطبيق مع معايير الـ **Accessibility**             |
| **UI Coverage**           | قياس نسبة تغطية واجهة المستخدم بالاختبارات                          |

---

### لماذا Cypress؟

الميزة الكبيرة في Cypress إنه يشتغل **داخل المتصفح مباشرة**، وهذا يعني:

- تنفيذ الأوامر في **Real-time** بدون تأخير
- رؤية كل خطوة في الـ **Test Runner** بشكل مرئي وواضح
- دعم قوي للـ **Debugging** مع **Time-Travel** داخل كل **Test Step**
- الحصول على **Insights** تساعدك تحسّن صحة الاختبارات وجودة التطبيق

### ملاحظة مهمة: كيف تحدد الـ Selectors？

Cypress يوفر أداة جاهزة داخل الـ **Test Runner** اسمها **Selector Playground**، تعطيك الـ **CSS Selector** أو الـ **Attribute Selector** لأي **Element** بدون ما تفتح الـ **DevTools**.

<div className="w-28">
  <OptimizedImage src="Cypress-Selector_rmyjbe" fullSize />
</div>

<br />

مثال على **Selector** يظهر تلقائيًا عند التحديد:

```js
cy.get('[name="username"]');
```

يظهر فوق مباشرة داخل الـ **Test Runner** وجاهز للنسخ بـ **Copy & Paste**.

هذه الأداة توفّر وقت كبير - بدل ما تبحث يدويًا في الـ **Inspect**، تنسخ الـ **Selector** في ثوانٍ.

**وهذا يعتبر أهم أداة تبدأ فيها كتابة الـ Test Cases بثقة وسرعة 👌🏻**

---

### كتابة Test Case للـ Login

```javascript
it.only("Visit And Login", () => {
  // Step 1: Navigate to the login page
  cy.visit(
    "https://opensource-demo.orangehrmlive.com/web/index.php/auth/login",
  );
  cy.wait(3000);

  // Step 2: Enter username and password
  cy.get('[name="username"]').type("Admin");
  cy.wait(3000);
  cy.get('[name="password"]').type("admin123");
  cy.wait(3000);

  // Step 3: Submit the login form
  cy.get('button[type="submit"]').click();
});
```

<br />

| الأمر        | الوصف                                                                  |
| ------------ | ---------------------------------------------------------------------- |
| `cy.visit()` | يفتح الـ **URL** المحدد في المتصفح                                     |
| `.type()`    | يُحاكي إدخال نص في حقل **Input**                                       |
| `.click()`   | يُحاكي الضغط على **Element** معيّن                                     |
| `cy.wait()`  | يُضيف تأخير زمني بالـ **Milliseconds** للانتظار حتى يكتمل تحميل الصفحة |

#### البحث بطريقة الـ Assertion مع الحذف

الـ **Assertion** هو التحقق البرمجي من وجود **Element** في الصفحة أو التأكد من قيمته.
نستخدمه للتحقق من وجود الموظف في الجدول قبل حذفه:

```javascript
// Step 1: Assert the employee exists in the table
cy.get(".oxd-table-body")
  .should("contain", "Almushdaq3")
  .then(() => cy.log("Employee verified successfully"));

cy.wait(4000);

// Step 2: Find the row and scroll into view
cy.contains(".oxd-table-row", "Almushdaq3")
  .scrollIntoView({ duration: 300 })
  .should("exist")
  .within(() => {
    // Step 3: Select the row checkbox
    cy.get(":nth-child(1) > .oxd-table-card-cell-checkbox .oxd-icon").click({
      force: true,
    });
    // Step 4: Click the delete icon
    cy.get(":nth-child(1) > .oxd-icon")
      .should("be.visible")
      .click({ force: true, multiple: true });
  });

cy.wait(1000);

// Step 5: Confirm deletion from the modal
cy.get(".orangehrm-modal-footer > .oxd-button--label-danger").click();
```

<br />

> استبدل `"Almushdaq3"` باسم الموظف اللي أنشأته في خطوة الإضافة.

## الـ Hooks في Cypress

### ما هي الـ Hooks؟

في الـ **Test Automation**، من أهم الأشياء إنك ما تكرر نفس الأكواد في كل **Test Case**.
الـ **Hooks** هي أكواد تُشغّلها Cypress **قبل** أو **بعد** الـ **Tests** بشكل تلقائي.

> **Hooks = أكواد تشتغل في وقت محدد بدون ما تستدعيها داخل كل Test.**

---

### أنواع الـ Hooks

| الـ Hook       | متى يشتغل؟                              | الاستخدام الأمثل                                   |
| -------------- | --------------------------------------- | -------------------------------------------------- |
| `before()`     | مرة واحدة قبل جميع الـ **Tests**        | الـ **Login** مرة واحدة، تجهيز **Test Data** ثابتة |
| `after()`      | مرة واحدة بعد انتهاء جميع الـ **Tests** | الـ **Logout**، تنظيف الـ **State**                |
| `beforeEach()` | قبل كل **Test Case**                    | فتح صفحة معيّنة، إعادة تهيئة الـ **State**         |
| `afterEach()`  | بعد كل **Test Case**                    | أخذ **Screenshot** عند الفشل، **Reset** للصفحة     |

---

### مثال واقعي

تخيّل عندك **5 Test Cases**:

1. تسجيل مستخدم
2. إضافة موظف
3. تعديل موظف
4. البحث عن موظف
5. حذف موظف

**بدون Hooks:** تكتب كود الـ **Login** خمس مرات - كود متكرر وصعب الصيانة.

**مع Hooks:** تحط الـ **Login** داخل `before()` مرة واحدة، وكل **Test** يبدأ مباشرة.

---

### مثال عملي بالكود

```javascript
describe("OrangeHRM - Employee Management Tests", () => {
  // Runs once before all tests - Login
  before(() => {
    cy.visit(
      "https://opensource-demo.orangehrmlive.com/web/index.php/auth/login",
    );
    cy.get('[name="username"]').type("Admin");
    cy.get('[name="password"]').type("admin123");
    cy.get('button[type="submit"]').click();
    cy.wait(3000);
  });

  // Runs before each test - Navigate to the target page
  beforeEach(() => {
    cy.visit(
      "https://opensource-demo.orangehrmlive.com/web/index.php/pim/viewEmployeeList",
    );
    cy.wait(2000);
  });

  it("Add new employee", () => {
    // Test logic here...
  });

  it("Search for employee", () => {
    // Test logic here...
  });

  it("Delete employee", () => {
    // Test logic here...
  });

  // Runs once after all tests
  after(() => {
    cy.log("All tests completed - session ended");
  });
});
```

<br />

| الجزء          | الوصف                                                                     |
| -------------- | ------------------------------------------------------------------------- |
| `before()`     | ينفّذ الـ **Login** مرة واحدة، جميع الـ **Tests** ترث نفس الـ **Session** |
| `beforeEach()` | يرجع للصفحة المطلوبة قبل كل **Test**                                      |
| `after()`      | ينفّذ بعد انتهاء كل شيء، مناسب للتنظيف أو الـ **Logging**                 |
| `describe()`   | يجمع الـ **Tests** تحت اسم واحد كـ **Test Suite** في الـ **Test Runner**  |

---

### متى تستخدم كل Hook؟

| السيناريو                            | الأنسب         |
| ------------------------------------ | -------------- |
| الـ **Login** مرة واحدة              | `before()`     |
| فتح صفحة معيّنة قبل كل اختبار        | `beforeEach()` |
| تنظيف بعد انتهاء كل شيء              | `after()`      |
| أخذ **Screenshot** عند فشل أي اختبار | `afterEach()`  |

> **الخلاصة:** الـ **Hooks** هي الطريقة الصحيحة لكتابة اختبارات نظيفة ومنظّمة، تقدر تصونها وتطوّرها بسهولة.

## الـ Regression Testing

### ما هو Regression Testing؟

> نعيد اختبار النظام بعد إضافة ميزة جديدة أو تعديل، عشان نتأكد إن الأشياء القديمة ما اتكسّرت.

لو فريق التطوير أضاف **Feature** جديد أو عدّل في جزء معيّن في الـ **frontend / backend**، نحن كـ **QA** ما نختبر الجديد فقط - نرجع نعيد اختبار كل شيء كان يشتغل قبله عشان نتأكد إنه **ما تكسّر** بسبب التغيير.

---

### كيف طبّقنا الـ Regression Testing؟

#### 1 - تنظيم ملفات الـ Test Cases

رتّبنا المشروع بحيث صار كل **Test Case** في ملف مستقل واضح وظيفته:

<div className="w-28">
  <OptimizedImage src="regression-testing1_qwe65r" fullSize />
</div>

<br />

الفكرة إني ما أكرر الأكواد - كل **Test Case** له ملف مستقل، وملف الـ **Regression** هو المسؤول عن تشغيلهم جميعًا بالـ **Order** الصحيح.

---

#### 2 - إنشاء ملف `Regression.cy.js`

هذا الملف يجمع كل الملفات السابقة ويعمل لها **Import بالترتيب**:

```javascript
/*
 * Regression Test Suite - Import Order:
 * 1. Visit Test
 * 2. Login Test
 * 3. Add Employee Test
 * 4. Search and Delete Employee Test
 * 5. Add Job Test
 * 6. Search and Delete Job Test
 * 7. My Info Test
 * 8. Logout Test
 *
 * Each test file should contain only one 'it' block.
 * The order of imports defines the order of execution.
 */

import "./loginAndVisitAndLogout/Visit.cy.js";
import "./loginAndVisitAndLogout/Login.cy.js";
import "./employee/AddEmp.cy.js";
import "./employee/SearchAndDeleteEmp.cy.js";
import "./job/AddJob.cy.js";
import "./job/SearchAndDeleteJob.cy.js";
import "./myInfo/MyInfo.cy.js";
import "./loginAndVisitAndLogout/Logout.cy.js";
```

---

### فكرة `testIsolation: false`

في `cypress.config.js` فعّلنا:

```js
testIsolation: false,
```

هذا الإعداد مهم لأنه يسمح بالانتقال من صفحة لصفحة في نفس الـ **Test Run** بدون ما يُعيد الصفحة من الصفر في كل **Test Case**.
يعني **باستمرارية الـ Session** - وهذا مناسب للـ **Regression** عشان نشغّل مجموعة **Tests** متتابعة بدون ما نرجع من البداية كل مرة.

## تصوير الشاشة والفيديو

Cypress يسمح لك تُصوّر الشاشة وتسجّل الفيديو أثناء تنفيذ الاختبارات، ويأخذ **Screenshots** تلقائيًا عند أي **Error**.

### لماذا هذا مهم؟

- **توثيق نتيجة كل Test Case** - دليل مرئي على كل خطوة
- **معرفة وش صار وقت الخطأ** - بدل ما تخمّن وين فشل الاختبار
- **مشاركة أدلة التنفيذ** مع الفريق بسهولة

---

### تفعيل تسجيل الفيديو

تصوير الفيديو تم تفعيله مع بداية تهيئة المشروع بإضافة:

```js
video: true,
```

لتشغيل الاختبارات مع التسجيل الكامل، استخدم الكوماند التالي:

```bash
npx cypress run --browser chrome --headed --spec "cypress/e2e/Regression.cy.js"
```

| الجزء              | الوصف                                    |
| ------------------ | ---------------------------------------- |
| `--browser chrome` | يشغّل الاختبار على متصفح **Chrome**      |
| `--headed`         | يفتح المتصفح بشكل مرئي (مو **Headless**) |
| `--spec`           | يحدد الملف المطلوب تشغيله                |

هذا الأمر يشغّل جميع الاختبارات داخل ملف **Regression**، يسجّل الفيديو، ويأخذ **Screenshots** تلقائيًا لو صار **Error**.

---

### كيف تأخذ Screenshot يدويًا؟

```js
cy.screenshot();
```

يلتقط صورة للشاشة في اللحظة التي أنت فيها.

**مثال عملي:**

```javascript
it("Add Employee", () => {
  // ... test steps

  // Take a screenshot after completing all steps
  cy.screenshot();
});
```

### ملاحظة: إذا بغيت تحتفظ بالفيديوهات السابقة

بشكل افتراضي Cypress يحذف الفيديوهات القديمة قبل كل تشغيل جديد.
لمنع هذا، أضف السطر التالي في `cypress.config.js`:

```js
module.exports = defineConfig({
  // Step 1: Enable video recording
  video: true,
  trashAssetsBeforeRuns: false, // Prevents deleting previous videos before new runs

  e2e: {
    setupNodeEvents(on, config) {
      // implement node event listeners here
    },
    // Step 2: Disable test isolation
    testIsolation: false,
  },
});
```

`trashAssetsBeforeRuns: false` يسمح بتراكم الفيديوهات بدون حذف المقاطع السابقة.
